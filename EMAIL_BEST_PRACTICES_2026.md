# HTML Email Best Practices 2026 for OSQ

Краткий рабочий конспект по результатам Tavily-поиска и сравнения с текущими OSQ-рассылками.

## Что уже хорошо работает в проекте

- Табличная верстка, центральный контейнер 600-640 px и крупные карточки совпадают с актуальной практикой.
- Основной контент держится живым HTML-текстом, а не запекается в баннеры. Это полезно для доступности, доставки и редактирования.
- Стиль последних писем OSQ уже близок к устойчивой модульной сборке: hero, продуктовые карточки, блоки преимуществ, CTA, footer.
- Паттерн из `2026-04-13-osq-email-case-bowl-2/Yandex Example.html` стоит использовать как инженерный ориентир: повторяемые ширины, радиусы, отступы, живой текст поверх визуала там, где это оправдано.

## Что нужно усилить

- Держать HTML ниже 102 KB, потому что Gmail обрезает письмо после этого порога. Целевой запас для новых писем: до 75-90 KB, если нет тяжелой юридической или промо-части.
- Не использовать больше одной тяжелой GIF в письме. Идеально держать GIF до 1 MB, но если клиент прислал важную анимацию, первый кадр должен быть полноценным статичным сообщением.
- Для GIF всегда проверять первый кадр: Outlook 2007-2019 на Windows и часть режимов экономии данных могут показать только его.
- Не полагаться на media queries как на основу мобильной версии. Для OSQ это особенно важно из-за мобильной Я.Почты: письмо должно сжиматься безопасно уже на desktop-каркасе.
- Использовать `role="presentation"` у layout-таблиц и нормальные `alt` у изображений. Декоративным иконкам можно оставлять пустой `alt=""`.
- CTA делать bulletproof: отдельная таблица, фон на `td`, ссылка `display:block`, `mso-padding-alt`, крупная зона клика.
- Для dark mode избегать чистых `#000000` и `#ffffff` как доминирующих фон/текст-пар, особенно в больших площадях. Лучше мягкие off-white, dark gray и достаточный контраст.
- Для логотипов и иконок предпочтительны прозрачные PNG с запасом контраста или отдельная темная версия, если фон может инвертироваться.
- Фоновые изображения с текстом поверх использовать только когда это реально нужно. Если блок критичен, добавлять Outlook/VML fallback; если нет, обычный `<img>` надежнее.

## Практические приемы для следующих OSQ-писем

- Делать письмо короче и модульнее: 1 hero, 3-5 главных продуктовых модулей, 1 блок события/CTA, footer.
- Для компиляций и дайджестов использовать принцип "витрина новинок": короткий текст, сильное фото, 2-3 тезиса, CTA. Не превращать письмо в каталог.
- Разделять роли изображений: hero/GIF для эмоции, продуктовые фото для доказательства, маленькие иконки для сканирования.
- Под выставки делать отдельный event-блок с датой, местом, стендом, промокодом и обещанием "что увидите вживую".
- Если нужна необычная механика, безопаснее всего работают: мини-бейджи, "маршрут продукта" через 3-4 шага, мягкие GIF/синемаграфы, кликабельные карточки, живой текст поверх email-safe фона.
- Для длинных компиляций полезен "сценарный" порядок: сначала приглашение и главный повод, затем 4-5 лучших новинок, затем технологическая демонстрация, затем финальный CTA.
