# Пайплайн рассылки OSQ (этапы 1–6)

Единая система для Claude Code и Codex. Все команды запускаются из корня проекта
`osq-email-assets`. Тексты пишутся ТОЛЬКО по скиллу `osq-voice`
(`.claude/skills/osq-voice/SKILL.md`), вёрстка — по контракту ниже и шаблону
`templates/email-skeleton.html`.

## Карта системы

| Что | Где |
|---|---|
| Скелет письма (канон v2) | `templates/email-skeleton.html` |
| Голос и тексты | `.claude/skills/osq-voice/` |
| Корпус стиля (регенерация) | `python scripts/extract_corpus.py` → `docs/corpus/` |
| Создать кампанию из архива | `python scripts/new_campaign.py <архив> <slug>` |
| Проверка письма | `python scripts/check_email.py <папка> [--online]` |
| Видео → mp4+GIF | `python scripts/video2gif.py <video>` |
| Деплой на GitHub + purge CDN | `python scripts/deploy.py` |
| Загрузка в Unisender | `python scripts/unisender_push.py …` |
| Секреты | `.env` (в git не попадает) |
| Общие ассеты (шрифты, лого, социконки) | `shared/fonts/`, `shared/images/` |

## Жёсткие константы письма

- Контейнер 640px (max), `min-width:320px`; карточки 620, радиус 32; внутр. поля 30.
- **HEADER и FOOTER — фиксированные**: структура из скелета, менять только
  utm_campaign, тексты не трогать. Логотип и социконки — из `shared/images/`.
- **HERO — фиксированный каркас**: картинка 620×390 background+VML, продукт
  в верхней части кадра, нижняя треть — под градиент
  `rgba(12,20,18,.86)→0` и живой текст (бейдж + H1 + подзаголовок).
- Картинки кроме HERO: **16:9 (620×349), 5:4 (620×496) или 1:1 (620×620)**;
  в двух колонках 270×152 / 270×216 / 270×270.
- Шрифты Qanelas из `shared/fonts/` (Regular 400, Medium 500, Bold 700, ExtraBold 800).

## Типографика v2 — «правило зума» мобильной Я.Почты

Мобильная Я.Почта НЕ применяет media queries: она сжимает десктоп 640→~360
(×0.5625). Поэтому минимальные ДЕСКТОПНЫЕ кегли:

| Элемент | px (desktop) |
|---|---|
| Основной текст | 19–20 / 27–28 |
| Лид под H1 | 22 / 28 |
| H2 блока | 34 / 40 |
| H1 hero | 40–44 |
| Текст кнопок | 22 bold, кнопка max-width 300px по центру |
| Бейдж hero | 16 caps bold |
| Подписи в карточках-статах | 16–17 bold |
| Текст в icon-карточках | 17 / 24 |
| Ссылки хедера | 16 |
| Реквизиты футера | 14 |
| Юр. сноски | 12 (минимум во всём письме) |

Меньше 16px смысловой текст НЕ делать. Абзацы короткие (2–4 предложения).
Максимум 2 колонки; в колонках только фото или «цифра + подпись».
Тест: открыть письмо в браузере при ширине 360px (сжатие) — всё читаемо?

## Этап 1 — приём материалов клиента

```
python scripts/new_campaign.py "<путь к архиву или папке>" <slug>
```
Создаст `YYYY-MM-DD-osq-email-<slug>/` с `ref/` (материалы клиента, в git
не попадают), `images/`, `index.html` (скелет с подставленными UTM/CDN)
и `BRIEF.md`.

## Этап 2 — изучение материалов

1. Инвентаризировать `ref/`: pptx/pdf/фото/рендеры.
2. Текст из pptx (слайды): `7z x файл.pptx -o<tmp> ppt/slides/*.xml` + вытащить
   `<a:t>`; либо скилл pptx. PDF — скилл pdf.
3. Просмотреть все картинки (Read) — что из этого пригодно для письма,
   какие рендеры станут референсами контейнеров для генерации.
4. Заполнить `BRIEF.md`: продукты и характеристики, инвентарь, план блоков,
   каких картинок не хватает (таблица генерации с референсами).
5. Показать пользователю краткий вывод: продукты, факты, план блоков, что будем
   генерить. Это НЕ жёсткий чекпойнт — можно сразу переходить к Этапу 3.

## Этап 3 — скелет письма (ЧЕКПОЙНТ пользователя)

1. Собрать `index.html` из блоков скелета: HERO + INTRO + FEATURE/SPECS/GALLERY/
   BENEFITS/TECH/ACCENT (по необходимости) + FINAL-CTA. Неиспользованные
   блоки-образцы удалить.
2. Тексты — полноценные, по `osq-voice` (профиль → лексикон → примеры →
   чеклист). Никакого lorem ipsum.
3. Картинки — серые плейсхолдеры с подписью и точным размером (уже в скелете).
4. Проверить: `python scripts/check_email.py <папка>` (плейсхолдеры на этом
   этапе допустимы) и открыть письмо в браузере (обычная ширина + 360px).
5. **СТОП: показать пользователю скелет и дождаться комментариев.**

## Этап 4 — картинки

Клиентские фото: кропнуть под форматы блоков (ffmpeg:
`ffmpeg -i src.jpg -vf "crop=W:H:x:y,scale=620:-1" out.jpg`), сжать до
разумного веса (<300KB продуктовые, hero <500KB), положить в `images/`.

Генерация — через скилл `image-gen` (модель gpt-image-2 по подписке ChatGPT):
```
node "%USERPROFILE%\.claude\skills\image-gen\scripts\codex-imagegen.mjs" "<промпт>" \
  --ref "<кампания>/ref/<рендер-контейнера>.png" --aspect landscape --out "<кампания>/images/<имя>.jpg"
```

**Правила промпта для контейнеров (КРИТИЧНО — контейнер нельзя деформировать):**
1. ВСЕГДА прикладывать `--ref` с чистым рендером именно этого контейнера из `ref/`.
2. В промпте прописать словами КАЖДУЮ деталь геометрии: форма и пропорции
   (квадратный/круглый/подушка), углы, борта, тип крышки (купольная PET,
   прозрачность) или её отсутствие, запайка (есть/нет плёнки), надписи и печать
   (обычно НЕТ надписей — писать «no printed text or logos on the container»),
   цвет и фактура (kraft cardboard texture / white / black wood).
3. Явно: «keep the exact container shape and proportions from the reference,
   do not deform, do not add or remove design elements».
4. Фон/сцена — мягкий натуральный стиль OSQ: деревянный стол, нейтральный фон,
   дневной свет, зелень; без людей и брендов.
5. После генерации СВЕРИТЬ с референсом (Read обеих картинок рядом): геометрия,
   крышка, борта, отсутствие лишних надписей. Деформация → итерация с уточнением.
6. Иконки: **только растровый PNG, SVG полностью запрещён** (и как файл, и как
   стиль генерации «3D»). Рецепт line-иконки: «flat minimal line icon, raster PNG,
   looks like a vector icon but is not SVG», штрих #333333 с акцентом #00A499,
   фон прозрачный (`--transparent`) или #F6F7F8, читаемость при 40px.

**Сцены — только светлые.** Клиент против тёмных/чёрных фонов и мрачных hero:
дневной свет, светлое дерево, нейтральный фон, зелень, «сочная» еда.

Hero-картинка: 620×390, продукт в верхней части кадра, нижние 35–45% кадра
свободны (туда ляжет градиент и текст). Остальные — только 16:9 / 5:4 / 1:1.

**Именование файлов** — семантическое: `<slug>-<блок>-<содержание>.jpg`
(например `pillow-hero-lifestyle.jpg`, `pillow-icon-lock.png`). Никаких
`май-01-02.jpg`. Парные карточки в письме делать одинаковой высоты
(атрибут height на td, как height="128" в эталоне), тексты в парах — одинакового
объёма.

После подготовки: `python scripts/deploy.py` → заменить плейсхолдеры на CDN-ссылки
→ `python scripts/check_email.py <папка> --online`.

## Этап 5 — видео и GIF (выборочно, по решению пользователя)

Генерация видео: Google VEO 3.1 Fast, image-to-video, **6 секунд, 720p, без
звука**. Пути (в порядке приоритета):
1. **MCP Magnific** (`mcp.magnific.com`, OAuth по аккаунту Freepik/Magnific,
   тратит кредиты подписки): инструмент `video_generate`, явно указать модель
   «Google Veo 3.1 Fast», приложить картинку-кадр. Если MCP не подключён —
   попросить пользователя: `claude mcp add --transport http magnific https://mcp.magnific.com`.
2. **REST API Magnific** (`POST https://api.freepik.com/v1/ai/image-to-video/veo-3-1-fast`,
   заголовок `x-freepik-api-key` из `.env` `FREEPIK_API_KEY`, тело:
   `{"image": "<https-url кадра>", "prompt": "...", "duration": 6,
   "resolution": "720p", "generate_audio": false}` → поллинг task_id). Внимание:
   API тарифицируется отдельно от подписки.
3. Браузерную автоматизацию сайта Freepik НЕ использовать (запрещена их AUP).

Дальше: скачать mp4 → `python scripts/video2gif.py <video.mp4>` (кроп 21:9 —
только по явному указанию пользователя: `--crop-21x9`) → переименовать по смыслу
→ положить `.gif` в `images/` кампании → `python scripts/deploy.py` → вставить
CDN-ссылку в письмо. GIF ≤1MB, первый кадр должен работать как статичный баннер.

## Этап 6 — публикация в Unisender

Нужен `UNISENDER_API_KEY` в `.env` (ЛК → Настройки → Интеграция и API).
```
python scripts/unisender_push.py lists                       # узнать list_id (один раз)
python scripts/unisender_push.py message <папка> --subject "Тема" \
    --list-id <id> --sender-email <подтверждённый адрес>     # письмо-черновик
python scripts/unisender_push.py test --message-id <id> --email <адрес>   # тест себе
```
Либо `template <папка> --subject "Тема"` — как шаблон (не требует list_id).
Скрипт сам заменит `{{BROWSER_URL}}`/`{{UNSUBSCRIBE_URL}}` на макросы
`{{ReadUrl}}`/`{{UnsubscribeUrl}}` и прогонит check_email.

## Финальный чеклист перед передачей пользователю

1. `python scripts/check_email.py <папка> --online` — 0 ошибок.
2. Чеклист текста из `osq-voice/references/checklist.md` пройден.
3. Письмо просмотрено в браузере на 640px и 360px.
4. Header/footer идентичны эталону, hero по канону.
5. Вес HTML ≤90KB, GIF ≤1MB (если есть).
6. ENG-версия (если нужна) — полный структурный эквивалент RU.
7. Ссылка «В браузере»: спросить у пользователя lovable-ссылку
   (`https://osq-mail-url.lovable.app/email/<uuid>`, отдельная для RU и ENG);
   если её нет — оставить макрос `{{BROWSER_URL}}` (unisender_push подставит
   `{{ReadUrl}}`).
8. По просьбе пользователя собрать архив всех финальных картинок письма
   (`7z a <slug>-final-images.zip <кампания>/images/*`).

## Известные грабли инфраструктуры

- jsDelivr кеширует файлы до 12 часов — после замены картинки с тем же именем
  обязателен purge (deploy.py делает сам). Новые имена файлов надёжнее замены.
- jsDelivr не отдаёт файлы >20MB и капризничает на гигантских репо — тяжёлые
  исходники держим в ref/ (не коммитится), в images/ кампании только
  оптимизированные файлы (фото <500KB, GIF <1MB).
- Вставка HTML в Unisender руками через веб-редактор ломала письма —
  использовать API-загрузку (`unisender_push.py`).
